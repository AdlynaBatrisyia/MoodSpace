📚 Lab Submissions

🎨
Lab 1 - Static UI with Jetpack Compose
Compose layouts · Column · Row · Box · Text · Image
Jetpack Compose Column Row Box Static UI

Built the first version of MoodSpace's home screen as a static UI using Jetpack Compose. I learned how to structure layouts using Column, Row, and Box, the three core layout composables. The screen included a top bar, daily goals section, and a bottom navigation bar, all styled to mirror the MoodFit reference app with a mint-green and sage colour palette.

Reflection: Lab 1 was my first time using Jetpack Compose and I found the declarative approach very different from XML layouts. The biggest challenge was understanding how Modifier chaining works, every layout decision flows through modifiers rather than separate attributes. By the end I could confidently build structured UIs and understood why Column, Row, and Box each serve a different layout purpose.
⚡
Lab 2 - Interactivity & State Management
TextField · Button · remember · mutableStateOf · Dialog
State TextField remember Dialog Interactivity

Added interactivity to MoodSpace by introducing state management with remember and mutableStateOf. Users could now enter their name, save it, and see the greeting update dynamically. I also added a Mood Dialog popup where users select their mood from 5 emoji options — Great, Good, OK, Poor, and Bad, and a Gratitude page with text input fields.

Reflection: This lab taught me the fundamental difference between stateless and stateful composables. I also learned the two-variable pattern, using one variable for live input and another for the saved display value, which prevents the UI from updating until the user confirms. The Mood Dialog was the most satisfying feature to build because it felt like a real app interaction.
🎨
Lab 3 - Material Design & Theming
MaterialTheme · Color.kt · Theme.kt · animateContentSize
Material Design Color Scheme Typography Animation Card

Applied Material Design 3 to MoodSpace by creating a custom Color.kt and Theme.kt with a mint-green primary palette and sage secondary colours. I wrapped all screens in MoodSpaceTheme and replaced plain containers with Card composables. I also added a spring-based animateContentSize animation to the Reminders card so it expands and collapses smoothly when tapped.

Reflection: Lab 3 showed me how much a consistent colour system improves the overall feel of an app. Defining colours once in Color.kt and referencing them everywhere made the UI much easier to maintain. The animateContentSize spring animation also taught me that great UX comes from small details, a single line of code made the reminder card feel polished and responsive.
🗺️
Lab 4 - Navigation & ViewModel
NavHost · NavController · sealed class · StateFlow · ViewModel
Navigation Compose ViewModel StateFlow Multi-Screen sealed class

Upgraded MoodSpace to a true multi-screen app using Jetpack Compose Navigation and ViewModel. I defined all routes in a sealed class called Screen and set up a NavHost with composable destinations for Home, Inbox, Inbox Detail, History, Insights, Sleep, and Gratitude. The MoodSpaceViewModel holds all shared state in a MoodSpaceUiState data class exposed via StateFlow, ensuring data persists across screen transitions.

Reflection: Lab 4 was the most complex lab so far and also the most rewarding. Understanding how ViewModel survives navigation while local state does not was a key insight. I also learned about dynamic route arguments — passing the message index through the route string to open specific inbox messages. This lab transformed MoodSpace from a single screen into a real application architecture.
🗄️
Lab 5 - Room Database Integration
Entity · DAO · RoomDatabase · Repository · StateFlow · MVVM
Room Entity DAO Repository MVVM Persistent Storage

Integrated Room Database into MoodSpace to make data persist after the app is closed. I created an ActivityEntity data class with @Entity annotation, an ActivityDao interface with insert, getAll, and delete operations returning Flow<List>, an AppDatabase singleton class, and an ActivityRepository connecting the ViewModel to the DAO. All mood, gratitude, sleep, and simple activity entries are now stored in a local SQLite database and verified through Android Studio's Database Inspector.

Reflection: Lab 5 completed the full MVVM + Repository + Room architecture pattern. The most important lesson was understanding why the Repository layer exists, it decouples the ViewModel from the data source, making the code testable and maintainable. Seeing data survive an app restart for the first time was genuinely exciting and made the purpose of persistent storage very clear.

Project 1 - MoodSpace (Lab 4 + Full Features)
SDG 3 · Multi-screen · ViewModel · Navigation Compose
SDG 3 Navigation ViewModel StateFlow 7 Screens

Problem Statement: Many young adults lack simple, accessible tools to monitor their mental well-being consistently. MoodSpace provides a structured daily check-in system for mood, gratitude, sleep, and personal activities, helping users build healthy mental health habits aligned with SDG 3.

Key features include: daily mood logging with 5 emoji options, a gratitude journal with 10 selectable prompts, sleep tracker with automatic duration calculation, simple activity logging, an inbox with 3 mental health messages, a history screen showing all entries, and a mental health assessment popup after completing all 3 daily goals that classifies the user's state and saves results to history.

📂 View on GitHub
Lessons Learned: Project 1 taught me how to tie together everything from Labs 1–4 into a complete, functional app. The biggest challenge was managing shared state across 7+ screens without data inconsistencies. I learned that a well-designed ViewModel with a single source of truth makes navigation seamless. I also learned the importance of the user flow, every screen should connect logically to the next.
🌐
Project 2 - MoodSpace Advanced (API + Sensor + Firebase + Room)
SDG 3 · Firebase Firestore · Retrofit API · Accelerometer · Room
SDG 3 Firebase Retrofit API Accelerometer Room 8 Screens

Problem Statement: Project 1 demonstrated daily mental health tracking, but data was lost on app close and the app had no connection to the outside world. Project 2 upgrades MoodSpace with persistent Room storage, Firebase Firestore cloud sync, live motivational quotes from the Quotable REST API, and an accelerometer-based breathing exercise feature triggered by shaking the device.

Four Technical Pillars:

  • Room Database — All mood, gratitude, sleep, activity, and mental health entries persist locally after app closes.
  • Firebase Firestore — Mental health check-in results are synced to the cloud after completing all daily goals.
  • Quotable REST API (Retrofit) — Daily motivational quotes are fetched live from api.quotable.io and displayed on a dedicated Quote screen.
  • Accelerometer Sensor — Shaking the device triggers a breathing exercise popup using the SensorEventListener to detect acceleration changes.
📂 View on GitHub
Lessons Learned: Project 2 was the most technically challenging work I have done in this course. Integrating four different systems, Room, Firebase, Retrofit, and the Accelerometer, each with their own threading and lifecycle requirements, taught me the importance of coroutines and viewModelScope for managing asynchronous operations cleanly. Firebase Firestore's real-time nature and Retrofit's suspend functions both rely on coroutines, which reinforced how central asynchronous programming is to modern Android development. I also learned that hardware sensors require careful lifecycle management, registering in onResume and unregistering in onPause to avoid battery drain.

💭 Reflection & Learning Journey

When I started Lab 1, I had never used Jetpack Compose and struggled to understand why layouts were written in Kotlin rather than XML. By Project 2, I was building multi-screen apps with database persistence, cloud integration, REST APIs, and hardware sensors, all within a single semester. The growth has been significant and sometimes overwhelming, but every lab built directly on the previous one in a way that made the progression feel natural.

The most important concept I learned throughout this course is the MVVM architecture, Model, View, ViewModel. Separating the UI from the business logic, and the business logic from the data source, made my code progressively cleaner and easier to debug. Lab 5's Repository pattern was the final piece that completed this architecture, and seeing it all work together in Project 2 was genuinely satisfying.

My chosen SDG : SDG 3, Good Health and Well-Being, motivated every design decision. Mental health is a topic that matters to me, and building an app that could genuinely help someone track their emotional well-being made the technical challenges feel worthwhile. From a simple static layout in Lab 1 to a fully connected, sensor-aware app in Project 2, MoodSpace has become something I am proud to have built from scratch.
